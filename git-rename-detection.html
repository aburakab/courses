Git Concepts

Git doesn't track renames.
It guesses them.

Git stores snapshots of files — not move operations. When you rename a file, Git later compares deleted files against new files and asks: does this look similar enough? Play with the detector below to see exactly how it decides.

01 · The big idea

Detection, not tracking

Git internally records what each file looked like at each commit — full snapshots, blob by blob. It does not store a "rename" operation. So when you ask Git to show what changed, it sees one file that vanished and one file that appeared, and runs rename detection on the pair.

03 · How Git decides similarity

It compares content reuse, not filenames

Git's actual algorithm hashes small chunks of each file and counts how many chunks appear in both. The score is roughly:

similarity score score = common_content / max(old_size, new_size)

That means a few important things:

  • The filename doesn't help. Only content overlap matters.
  • Size matters. Tiny files fall below the threshold easily — change 3 lines of a 5-line file and you're in D+A territory.
  • Reformatting hurts. Changing indentation, line endings, or wrapping can tank the score even when the logic is identical.
04 · The threshold lever — -M

You can tighten or loosen the rule

Git's default rename threshold is 50%, and since Git 2.9 rename detection is already on by default. The -M<n>% flag mainly lets you override that threshold per command.

Force detection at default 50%git diff -M
Same, long formgit diff --find-renames
Strict — only count ≥ 90% similargit diff -M90%
Loose — count anything ≥ 20% similargit diff -M20%
Detect renames in staged changesgit diff --cached -M
Show rename status (R### codes)git diff --cached --name-status -M

Example output of --name-status:

R085    src/greeting.cpp    utils/messages.cpp
D       src/greeting.cpp
A       utils/messages.cpp
05 · Why VS Code sometimes shows "deleted + added"

Your editor just repeats what Git says

VS Code shows you whatever Git reports. If Git couldn't detect the rename — because the similarity dropped below the threshold — VS Code will show one deleted file and one added file. Even if you know it was a rename, your editor doesn't, because Git itself doesn't.

06 · Best practice — split the move from the edit

Two commits, clean history

If you have to rename and heavily change a file, do it in two commits. Git will detect the pure move as R100, and the second commit shows only the real logic change.

commit 1 · the move Move greeting file to utils folder 
git mv src/greeting.cpp utils/messages.cpp
git commit -m "Move greeting file to utils folder"
then →
commit 2 · the edit Update greeting message logic 
# edit utils/messages.cpp …
git commit -am "Update greeting message logic"
07 · The mental model

Git is a teacher comparing two papers

One paper disappeared. One new paper appeared. The teacher squints at both and asks:

"Same paper, new name." renamed

Enough of the writing is shared that it's obviously a continuation of the old paper. Git records it as a rename.

"Old paper trashed, new one written." deleted + added

The new paper shares too little with the old one to count. Git treats them as unrelated files.

The rule: Git does not know you renamed it — it guesses by measuring similarity.
